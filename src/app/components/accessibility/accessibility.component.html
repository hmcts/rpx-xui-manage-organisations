Accessibility statement for MyHMCTS Manage Organisation

MyHMCTS Manage Organisation is used across several services. Please use links to find contact information for each service.

  • Immigration and Asylum
  • Family Public Law
  • Adoption
  • Civil Damages
  • Divorce and Financial Remedies
  • Civil Money Claims
  • Private Law
  • Probate
  • Social Security and Child Support

This website is run by HM Courts & Tribunals Service. We want as many people as possible to be able to use this website. For example, you should be able to:

  • change colours, contrast levels and fonts using browser or device settings
  • navigate most of the website using a keyboard or speech recognition software
  • listen to most of the website using a screen reader (including the most recent versions of JAWS, NVDA and VoiceOver)

We’ve also made the website text as simple as possible to understand.

AbilityNet has advice on making your device easier to use if you have a disability.

How accessible this website is

We know some parts of the website are not fully accessible:

  • some pages can't be zoomed in to 400% without content overlapping
  • some page titles aren't page specific
  • some pages do not follow a logical tabbing order
  • some headings levels are skipped
  • some links do not work
  • some PDFs may not be fully accessible to a screen reader

Technical information about this website’s accessibility

HMCTS is committed to making its website accessible, in accordance with the Public Sector Bodies (Websites and Mobile Applications) (No.2) Accessibility Regulations 2018.

Compliance Status

This website is partially compliant with the Web Content Accessibility Guidelines version 2.1 due to the non-compliances listed below.

Non-accessible content

The content listed below is non-compliant for the following reasons.

Non-compliance with the accessibility regulations

There were navigation lists present that were not marked up correctly and so assistive technology may have difficulty reading the content. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A) and 4.1.2 Name, Role, Value (Level A).

For screen reader users, it will be difficult for the user to distinguish which form elements are related to each document as they don’t have a shared or common label. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A) and 3.3.2 Labels or Instructions (Level A).

Links were found that did not have a distinct meaning which could mean that screen reader users may be unable to determine their purpose. This fails WCAG v2.2, 2.4.4 Link Purpose (In Context) (Level A) and 2.4.9 Link Purpose (Link Only) (Level AAA).

A string of text that visually named an element was not part of the programmatic accessible name. When using assistive technology, it may be problematic for the user to interact with the element using their voice. This fails WCAG v2.2, 2.5.3 Label in Name (Level A).

A submit button is not provided for a text input element so entering text into the input used to search documents causes the document’s context to change automatically, which can be disruptive for assistive technology users. This fails WCAG v2.2, 3.2.2 On Input (Level A).

Visual headings were found, but they were not marked up with a semantic heading tag. Using assistive technology, a screen reader user may not be aware that the text is a heading, which means they may struggle to navigate using headings and understand the correct hierarchy of the content. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A).

Active elements are nested, which can cause random results from assistive technology.  This means users might be unable to accurately interact with the content or they could be missing content. This fails WCAG v2.2, 4.1.2 Name, Role, Value (Level A).

The state of content that can be expanded or collapsed is not announced or not correctly announced to screen reader users; this means screen reading software will only announce a collapsed state. This fails WCAG v2.2, 4.1.2 Name, Role, Value (Level A).

Asterisks are used to denote a required field. The asterisk is not clearly communicated to enable a user to understand that it denotes a required field. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A) and 3.3.2 Labels or Instructions (Level A).

Some headings are contained within a description list, which may cause some confusion to a screen reader user. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A).

When a user chooses an option from a search field, the chosen option will be added to the service webpage. Users can remove the options by activating an interactive element. When using assistive technology, the user may not know that the new interactive element is used to remove the added option as it does not have a descriptive alternative. This fails WCAG v2.2 1.3.1 Info and Relationships (Level A).

Typing into the text input causes a list box to appear and show options that the user can choose from. If the user is using keyboard inputs to navigate the service webpage, the user will have difficulties selecting an option. This fails WCAG v2.2, 2.1.1 Keyboard (Level A) and 2.1.3 Keyboard (No Exception) (Level AAA).

Using mouse inputs to activate the tab elements will cause the service webpage contents to change. It is not possible for a keyboard-only user to navigate or activate the tabs. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A), 2.1.1 Keyboard (Level A), 4.1.2 Name, Role, Value Level (A), and 2.1.3 Keyboard (No Exception) (Level AAA).

Headings on the page were found to not follow a hierarchical heading structure, which means assistive technology may struggle to communicate a logical heading order to the user. This fails WCAG v2.2, 1.3.1 Info and Relationships (Level A).

Some images were decorative, but the descriptions were not accurate for assistive technology users. This fails WCAG v2.2 1.1.1 Non-text Content (Level A).

Elements are present that do not contain a descriptive name or label for users. Assistive technology users may not be able to accurately identify their differences. This fails WCAG v2.2, 2.4.6 Headings and Labels (Level AA).

Assistive technology does not announce the important messages shown if there are no results, and this means that a screen reader user might not know that there are no results available. This fails WCAG v2.2, 4.1.3 Status Messages (Level AA).

Some of the colour combinations found on the website are low contrast and are likely to be difficult for people with low vision to distinguish. This fails WCAG v2.2, 1.4.11 Non-text Contrast (Level AA).

Content is lost or obscured on the page when the browser dimensions are set to 1280 x 1024 and magnified to 400. The user will need to modify their settings, and this could make it difficult to read the service webpage. This fails WCAG v2.2, 1.4.10 Reflow (Level AA).

Text on the website cannot be resized to 200% without loss of content/ functionality. This fails WCAG v2.2, 1.4.4 Resize text (Level AA).

Some of the text colour combinations found on the site do not meet contrast guidelines. The change in colour may make it difficult for the user to read the text as the contrast is less than the minimum required. This fails WCAG v2.2, 1.4.3 Contrast (Minimum) (Level AA) and 1.4.6 Contrast (Enhanced) (Level AAA).

Modifying the text spacing to make the text easier to read will cause the ‘Zoom in’ button to be obscured. The user will need to change their settings, and this could make the text difficult to read. This fails WCAG v2.2, 1.4.12 Text Spacing (Level AA).

Abbreviations are not expanded in the first instance or explained and may be difficult for users with cognitive impairments to understand. This fails WCAG v2.2 3.1.4 Abbreviations (Level AAA).

There are elements which open in a new window without informing the users first. If the user is not aware a new window/ tab will open, the user may be disoriented and have difficulties returning to the previous page. This fails WCAG v2.2 3.2.5 Change on Request (Level AAA).

Some links were not descriptive enough for assistive technology to convey the context to the user. This fails WCAG v2.2 2.4.9 Link Purpose (Link Only) (Level AAA).

Text colour combinations found on the service do not meet enhanced contrasts and are likely to be difficult for people with vision colour deficiencies to read. This fails WCAG v2.2, 1.4.6 Contrast (Enhanced) (Level AAA).

Disproportionate burden on accessibility regulations

The following areas have been classified as a disproportionate burden within the meaning of the accessibility regulations. These issues are:

  • Screen readers not picking up how pages are related to one another, and where they are in the context of the whole site
  • Screen readers not picking up on non-unique table headings on the same page
  • Some labels, fields, error messages and other content not being clearly identified by screen readers and voice activation software on mobile devices.
  • There are areas which have been classified as disproportionate burden. These include zooming in 200% on mobiles and more widely on navigation as detailed in WCAG 2.1 success criterion 2.4.5: Multiple Ways (Level AA).

Content that’s not within the scope of the accessibility regulations

PDFs, documents and attachments to email notifications published before 23 September 2018 may not meet accessibility standards. For example, they may not be structured so they’re accessible to a screen reader.

Any new PDFs or Word documents we publish will meet accessibility standards.

What we’re doing to improve accessibility

We’re committed to ensuring our services are accessible to all our users and that they comply with level AA of the Web Content Accessibility Guidelines – WCAG 2.2.

We’re actively working to improve the identified issues and make them AA-compliant by December 2024.
